Primary demo document:
- Use `User's Guide for the Commercial Modular Aero-Propulsion System Simulation (C-MAPSS)` as the example document for the demo.

Overall goal:
- Build a practical end-to-end demo that works with minimal setup.

- Do not overengineer the project.
- Explicitly demonstrate how RAG reduces hallucinations compared to a pure LLM approach.

ANTI-HALLUCINATION REQUIREMENTS (CRITICAL)
----------------------------------------

- The system must explicitly minimize hallucinations.
- The LLM must be instructed with a strict grounding policy:

  "Answer ONLY using the provided retrieved context. 
   If the answer is not contained in the context, say:
   'I cannot find this information in the document.' 
   Do NOT fabricate or infer missing information."

- This instruction must always be included in the prompt to the LLM.
- This instruction must be included ONLY in RAG answer generation prompts.
- Do NOT include this strict grounding instruction in LLM-only prompts.
- The app must clearly communicate to the user when:
  - the answer is grounded in retrieved context
  - or when the system cannot find the answer

USER WORKFLOW & FLOW CLARITY (CRITICAL)
----------------------------------------

The app must have a clear workflow that separates preparation from querying:

1. Document Preparation (one-time per document):
   - User selects PDF source: file path or upload.
   - User clicks "Index Document" button explicitly.
   - App shows progress: Extracting text... → Chunking... → Building embeddings... → Ready.
   - State saved to `.cache/` with persistent indicator: "Document indexed: ✓ C-MAPSS (timestamp)".
   - Once indexed, repeated queries reuse the cached index without re-indexing.

2. Query Execution (after indexing):
   - User enters question in text input field.
   - User presses Enter OR clicks "Query" button.
   - App retrieves top-5 chunks from cached index (instant, no indexing).
   - App displays retrieved chunks and generates answer.
   - User can submit multiple queries without re-indexing.

3. Mode Awareness:
   - Display active mode (RAG or LLM-only) prominently before each answer.
   - If switching documents, trigger re-indexing if document hash differs.
   - Show "Index ready" status after first indexing completes.

Reasons this matters:
  - Clarity: Users understand that indexing happens once, queries are fast.
  - UX: Separates slow operation (indexing) from fast operation (querying).
  - Performance expectation: Justifies why first interaction takes longer.
  - Trust: Visible caching state shows work is not wasted.

LLM-ONLY MODE GUIDE
----------------------------------------

LLM-only mode MUST be clearly explained in UI and README:

Purpose:
  - Demonstrate difference between RAG (grounded) and pure LLM (ungrounded).
  - Show how RAG reduces hallucinations by comparison.
  - Educational: let users see LLM tendencies without retrieval context.

When LLM-only is DISABLED:
  - No valid Gemini API key configured.
  - Show message: "LLM-only requires a valid Gemini API key."
  - Only RAG mode (retrieval-only fallback) is available.

When LLM-only is ENABLED (valid key):
  - User can toggle between RAG and LLM-only modes.
  - Clear label: "LLM-only Mode (may hallucinate)"
  - Behavior: send question directly to Gemini WITHOUT retrieval context.
  - Result: answer grounded=false, no Retrieved Chunks shown for clarity.
  - Optional comparison: checkbox "Show RAG vs LLM-only side-by-side".

Important UI notes for LLM-only:

  ⚠️  CRITICAL IMPLEMENTATION WARNING — READ BEFORE CODING:

  The `generate_llm_only_answer` function MUST use a completely separate prompt
  from `generate_rag_answer`. These two functions have OPPOSITE grounding goals:

  generate_rag_answer   → STRICT grounding policy (answer ONLY from context)
  generate_llm_only_answer → FREE answer (answer from general knowledge, NO context)

  DO NOT copy or reuse STRICT_GROUNDING_POLICY in the LLM-only prompt.
  If you do, Gemini will always respond:
      "I cannot find this information in the document."
  because there is no retrieved context to ground on — which breaks the demo
  completely and makes LLM-only mode indistinguishable from a broken state.

  Correct LLM-only prompt template:
      "You are a helpful assistant. Answer the following question using your
       general knowledge. No document context is provided — this is intentional.
       Answer as completely and clearly as you can.

       Question: {question}"

  NEVER add to LLM-only prompt:
      - STRICT_GROUNDING_POLICY
      - "Answer ONLY using the provided retrieved context."
      - "If the answer is not contained in the context, say: ..."
      - Any reference to retrieved context or document grounding.

  - Do NOT retrieve and display chunks in LLM-only mode (no retrieval occurs).
  - Show clear badge: "LLM-only Mode (may hallucinate)".
  - In comparison view, show LLM-only answer alongside RAG answer for contrast.

README guidance on LLM-only:
  - Explain use case: educational comparison, hallucination demonstration.
  - Show example: question where LLM hallucinates vs RAG grounds it.
  - Warn: LLM-only has no document constraints; answers may be unreliable.
  - Recommend: use RAG mode for real document QA, LLM-only for learning.

CACHING & PERSISTENCE VISIBILITY
----------------------------------------

Cache behavior must be visible and understandable to users:

- Stored in `.cache/` folder with persistent naming:
  - `chunks_{source_id}.json` — extracted and chunked text
  - `metadata_{source_id}.json` — chunk metadata (page, section, token count)
  - `faiss_{source_id}.index` — semantic search index
  - `embeddings_{source_id}.npy` — embedding vectors
  - `uploads/{source_id}.pdf` — uploaded PDFs (if using Upload mode)

- Source ID is SHA256 hash of PDF bytes (first 16 chars), ensuring identical PDFs reuse cache.

UI indicators:
  - After first indexing: show "✓ Document indexed: <full_path> (<N> chunks, indexed at <timestamp>)"
    Example: "✓ Document indexed: data/document.pdf (86 chunks, indexed at 2026-04-02 15:56:27)"
  - On subsequent sessions: show "✓ Loaded from cache: <full_path> (<N> chunks, indexed at <timestamp>)"
  - Use the full path (e.g. data/document.pdf), NOT just the filename.
    Reason: user may have PDFs in different folders; full path avoids ambiguity.
  - If document changes: automatically reindex and update cache.
  - Show cache state in UI regardless of source (file path or upload).

Caching benefits:
  - Cold start (first run): 20-45 seconds for typical 100-page PDF.
  - Warm start (same PDF, new session): 5 seconds (load chunks from JSON + embeddings from disk).
  - Subsequent queries in same session: instant (already in memory).

User should understand:
  - First query with new PDF is slow: computing embeddings.
  - Second session with same PDF: faster, loading from `.cache/`.
  - Switching documents: will reindex only if different from cached.

GEMINI TEMPERATURE & MODE TAXONOMY (CRITICAL)
----------------------------------------

The app must expose Gemini temperature controls and clearly distinguish operational cases.

Temperature control requirements:
- Add a Gemini temperature control in the sidebar.
- Include quick preset buttons:
  - `T=0.0`
  - `T=0.2`
  - `T=0.7`
- Include a slider for fine-grained control:
  - min: 0.0
  - max: 1.5
  - step: 0.1
- Apply the selected temperature to ALL Gemini generation calls:
  - RAG+Gemini answer generation
  - LLM-only answer generation

Mode taxonomy to display in UI:
1. RAG-puro (retrieval-only, no Gemini)
   - Trigger: no valid Gemini API key OR Gemini provider failure
   - Behavior: retrieval-only answer from top evidence/snippet template

2. RAG+Gemini deterministic (T=0.00)
   - Trigger: RAG mode + valid Gemini + temperature == 0
   - Behavior: grounded answer generation with lower output variability

3. RAG+Gemini stochastic (T>0)
   - Trigger: RAG mode + valid Gemini + temperature > 0
   - Behavior: grounded answer generation with higher output variability

4. Gemini-only deterministic (T=0.00)
   - Trigger: LLM-only mode + valid Gemini + temperature == 0
   - Behavior: ungrounded answer generation, lower variability

5. Gemini-only stochastic (T>0)
   - Trigger: LLM-only mode + valid Gemini + temperature > 0
   - Behavior: ungrounded answer generation, higher variability

Comparison panel guidance:
- Primary recommended comparison (for reproducibility):
  - Left: RAG+Gemini deterministic (T=0)
  - Right: Gemini-only deterministic (T=0)
- Optional secondary comparison (for creativity/variance demos):
  - Compare both with the same T>0 value (example: T=0.7)

Labeling requirements:
- Every answer panel must show an explicit mode label including temperature.
- Suggested labels:
  - `RAG-puro (retrieval-only, sin Gemini)`
  - `RAG+Gemini determinista (grounded answers, T=0.00)`
  - `RAG+Gemini (grounded answers, T={value})`
  - `Gemini-puro (LLM-only, T={value})`

Grounding semantics clarification:
- Grounded: answer constrained to retrieved document context.
- Ungrounded: answer generated from general model knowledge without retrieval context.
